# ShopGuide Backend Client Integration Guide

## 1. 后端在做什么

ShopGuide 后端是一个 FastAPI 服务，负责：

- 理解用户购物需求；
- 抽取类目、预算、排除条件、场景偏好；
- 从真实商品库做 RAG 检索；
- 执行硬过滤，比如不含酒精、不要日系品牌、预算上限；
- 对候选商品排序、rerank，并让 LLM 在候选池内做最终商品选择；
- 后端再次校验 LLM 选择，动态输出 0-4 张商品卡；
- LLM 不再有 legacy planner；当前只用于语义理解、候选池内商品选择、基于 evidence 的自然语言表达。
- Recommendation memory 命中后会复用结构化选品结果，跳过重复检索、排序和 LLM 商品选择；命中结果仍会重新校验硬约束。
- 返回可解释推荐、商品卡片、快捷修正按钮；
- 维护 session 上下文、商品追问、自然语言购物车和模拟下单。

客户端主要负责输入、流式渲染、商品卡片展示、BottomSheet 追问、购物车 UI 和用户操作。

## 2. 代码结构

核心代码在 `backend/app`：

```text
main.py                FastAPI HTTP / WebSocket 入口
agent.py               导购 Agent 主流程，编排推荐、LLM selection、追问、对比、bundle、购物车
planner_agent.py       兼容规则工具，提供部分 guard/debug plan helper；不调用 LLM
constraint_filter.py   硬约束过滤
embedding_retriever.py BM25 / embedding 检索
ranker.py              分层排序，生成候选商品、推荐理由和证据
cart.py                购物车增删改查和模拟下单
models.py              请求、商品、检索计划、Session 等数据结构
tts_adapter.py         TTS 预留适配层，目前是 stub
```

商品数据来自：

```text
ecommerce_agent_dataset/*/data/*.json
```

## 3. 启动后端

在服务器上：

```bash
ssh mix_A100
cd /home/huadabioa/houlong/SoulDance
```

当前项目拆成两个 Python 环境：

```text
env/venv_shopguide_backend  FastAPI + RAG + tests + smoke demo
env/venv_vllm_cu128         Qwen3-TTS / vLLM-Omni only
```

不要把后端依赖装进 `env/venv_vllm_cu128`，它是 TTS/vLLM-Omni 专用环境。

首次创建后端环境：

```bash
bash server/scripts/setup_backend_env.sh
```

先跑测试：

```bash
cd server && ../env/venv_shopguide_backend/bin/python -m pytest tests/test_agent_core.py tests/test_api.py -v
```

启动服务：

```bash
export ARK_API_KEY="从安全位置填入，不要写进代码"
export ARK_BASE_URL="https://ark.cn-beijing.volces.com/api/v3/"
export ARK_MODEL="ep-20260514111645-lmgt2"
export EMBEDDING_MODEL_DIR="model/bge-small-zh-v1.5"

ARK_API_KEY="$ARK_API_KEY" bash server/scripts/start_backend.sh
```

健康检查：

```bash
curl http://127.0.0.1:18080/health
```

正常会返回：

```json
{
  "status": "ok",
  "product_count": 100,
  "llm": "doubao",
  "retriever": "embedding"
}
```

如果没有设置 `ARK_API_KEY`，后端会使用 `FakeLLMClient`，适合本地调试接口形态，但真实回复效果会变弱。

## 5. WebSocket 主接口

聊天主流程使用：

```text
WS /ws/chat
```

### 5.1 普通推荐

客户端发送：

```json
{
  "type": "user_message",
  "session_id": "demo_session_001",
  "message": "推荐防晒霜，但不要含酒精的，也不要日系品牌",
  "input_type": "text",
  "tts_enabled": false
}
```

### 5.2 商品详情追问

用户点开商品卡片后，在 BottomSheet 里继续追问：

```json
{
  "type": "product_followup",
  "session_id": "demo_session_001",
  "focus_product_id": "p_beauty_xxx",
  "message": "这个有点贵，有没有100以内的？",
  "tts_enabled": false
}
```

`focus_product_id` 必须来自前面 `product_item.product.product_id`。

### 5.3 显式购物车操作

```json
{
  "type": "cart_action",
  "session_id": "demo_session_001",
  "action": "add_to_cart",
  "product_id": "p_beauty_xxx",
  "quantity": 1
}
```

支持的 `action`：

```text
add_to_cart
update_quantity
remove
checkout
```

### 5.4 自然语言购物车操作

也可以直接发普通消息：

```json
{
  "type": "user_message",
  "session_id": "demo_session_001",
  "message": "把刚才那款加到购物车"
}
```

后端会基于 session 最近推荐和当前商品焦点解析商品引用。

## 6. WebSocket 返回事件

普通推荐的事件顺序：

```text
assistant_state
text_delta
assistant_state
text_delta
products_start
product_item...
products_done
quick_actions
done
```

第二个 `assistant_state` 通常表示 LLM 商品选择阶段，可能包含：

```json
{
  "selection_mode": "llm_selection",
  "candidate_count": 6,
  "selected_count": 2
}
```

客户端不用自己决定推荐数量，收到几条 `product_item` 就渲染几张卡。当前后端上限是 4 张；低相关或无匹配时可以是 0 或 1 张，不会为了凑满固定数量强推无关商品。
当前协议里 `product_item` 不是 RAG 候选，而是后端完成 LLM selection、硬约束复核和解释文本之后的最终展示卡片。

客户端建议处理方式：

| 事件 | 客户端动作 |
|---|---|
| `assistant_state` | 显示轻量状态，例如正在筛选 |
| `text_delta` | 追加到当前 AI 气泡 |
| `products_start` | 显示商品骨架屏 |
| `product_item` | 插入商品卡片 |
| `products_done` | 停止商品骨架屏 |
| `quick_actions` | 渲染快捷修正按钮 |
| `replacement_product` | 兼容商品详情页追问里的替代商品；普通聊天换款仍会收到标准 `product_item` |
| `clarification_request` | 显示澄清问题和选项 |
| `comparison_result` | 显示商品对比结果；兼容旧字段，并可读取 `dimensions`、`items[].dimension_values`、`items[].risk_flags` 做结构化对比 |
| `bundle_start` / `bundle_item` / `bundle_done` | 显示场景组合推荐 |
| `filter_recovery_options` | 无匹配时显示放宽条件 |
| `cart_update` | 更新购物车 badge 和购物车页面 |
| `done` / `focus_done` | 当前轮结束 |
| `error` | 展示错误提示，不要让 App 崩溃 |

旧客户端可以忽略不认识的事件，只处理 `text_delta`、`product_item`、`replacement_product`、`cart_update` 和结束事件。普通聊天里的换款 follow-up 会继续发标准 `product_item`，不用依赖 `replacement_product` 才能看到卡片。

### 6.1 商品卡准入规则

商品卡不是检索完就直接发。后端当前流程是：

```text
LLM 意图识别
后端购物准入校验
taxonomy / 硬过滤
ranker / reranker 候选池
LLM selection_decision
后端最终校验
LLM 解释文本
product_item
```

这意味着：

- `你好`、`你是谁`、`我是猪`、乱码等非购物输入不出商品卡；
- `推荐毛巾` 是购物输入，但如果数据集没有毛巾 taxonomy，走 `filter_recovery_options`，不乱推商品；
- LLM 只能从候选池里选已有商品 ID，不能编造商品；
- 预算、品牌、类目、成分等硬约束会在 LLM 选择后再检查一次。
- 同一 session 里明确提出新类目会开启新购物任务；只回答预算/偏好时才继承上一轮澄清问题。
- 澄清按钮只补偏好，不替用户新增预算、品牌、成分等硬约束；只有用户明确输入预算时才写入价格约束。

### 6.2 商品卡片事件示例

```json
{
  "type": "product_item",
  "message_id": "assistant_xxx",
  "index": 0,
  "role": "primary",
  "product": {
    "product_id": "p_beauty_xxx",
    "name": "清爽防晒乳",
    "brand": "示例品牌",
    "category": "美妆护肤",
    "sub_category": "防晒",
    "price": 129.0,
    "main_image_url": "...",
    "tags": ["美妆护肤", "防晒", "中国"],
    "reason": "类目精确匹配防晒，匹配清爽"
  }
}
```

`role=primary` 是主推商品，客户端应该更突出展示。其它商品是备选。正式商品卡不再返回原始 `evidence`；证据只在后端内部用于选择、解释和用户主动追问。

### 6.3 快捷修正事件示例

```json
{
  "type": "quick_actions",
  "message_id": "assistant_xxx",
  "actions": [
    {
      "label": "更便宜",
      "message": "换个更便宜的",
      "payload": {
        "focus_product_id": "p_beauty_xxx"
      }
    },
    {
      "label": "不要示例品牌",
      "message": "不要示例品牌，换一款",
      "payload": {
        "focus_product_id": "p_beauty_xxx"
      }
    }
  ]
}
```

客户端点击后：

- 如果 `payload.focus_product_id` 存在，建议发 `product_followup`；
- 否则发普通 `user_message`；
- `message` 就是下一轮发送给后端的用户文本。

## 7. HTTP 接口

### 7.1 健康检查

```text
GET /health
```

### 7.2 商品列表

```text
GET /api/products?limit=20
```

返回商品摘要，适合客户端调试商品卡片。

### 7.3 商品详情

```text
GET /api/products/{product_id}
```

返回完整商品信息。

### 7.4 调试检索计划

```text
POST /api/debug/retrieval_plan
```

请求：

```json
{
  "type": "user_message",
  "session_id": "demo",
  "message": "推荐一款适合油皮的洗面奶，预算100以内"
}
```

返回 `RetrievalPlan`，方便联调意图识别、预算、类目、排除条件。

### 7.5 购物车 REST

```text
GET  /api/cart?session_id=demo_session_001
POST /api/cart/add
POST /api/cart/update_quantity
POST /api/cart/remove
POST /api/cart/clear
POST /api/cart/checkout
```

POST body：

```json
{
  "session_id": "demo_session_001",
  "product_id": "p_beauty_xxx",
  "quantity": 2
}
```

`checkout` 会模拟下单并清空购物车。

## 8. 推荐客户端接入流程

1. App 启动时生成并保存一个稳定的 `session_id`。
2. 建立 `WS /ws/chat` 连接。
3. 用户输入时发送 `user_message`。
4. 收到 `text_delta` 追加 AI 气泡。
5. 收到 `product_item` 渲染商品卡片。
6. 用户点商品卡片时打开 BottomSheet，并保存 `product_id`。
7. BottomSheet 里继续问时发送 `product_followup`。
8. 点击 `quick_actions` 时，把 action 的 `message` 发回后端。
9. 收到 `cart_update` 时刷新购物车 badge、金额和商品列表。
10. 收到 `done` 或 `focus_done` 后结束当前轮 loading。

## 9. 语音能力（STT + TTS）

### 9.1 STT：语音输入

App 端语音输入流程：

1. 使用 `AudioRecord` 录制 PCM 16bit mono 16kHz 音频。
2. 封装为 WAV（44 字节头）。
3. 调用 `POST /api/stt`，multipart 上传音频文件：

```bash
curl -X POST http://<host>:<port>/api/stt \
  -F audio=@recording.wav \
  -F session_id=<session_id> \
  -F audio_format=wav
```

返回示例：

```json
{
  "text": "推荐防晒霜",
  "is_final": true,
  "confidence": 0.92,
  "language": "zh"
}
```

4. 将返回的 `text` 显示在输入框中（或自动发送），并通过 WebSocket 发送：

```json
{
  "type": "user_message",
  "session_id": "...",
  "message": "推荐防晒霜",
  "input_type": "voice",
  "tts_enabled": true,
  "voice": "calm_female"
}
```

`input_type` 可选值为 `"text"`（默认）或 `"voice"`。

### 9.2 TTS：语音播报

发送消息时携带 `tts_enabled: true` 和 `voice`，后端在生成文本回复后会通过同一 WebSocket 连接下发音频事件：

- `audio_delta`：Base64 编码的 WAV 音频切片。

```json
{
  "type": "audio_delta",
  "message_id": "assistant_xxx",
  "encoding": "wav",
  "sample_rate": 24000,
  "channels": 1,
  "data": "<base64 encoded bytes>",
  "is_last": false
}
```

- `audio_done`：当前句子播报结束。

```json
{
  "type": "audio_done",
  "message_id": "assistant_xxx"
}
```

- `audio_error`：语音合成失败（可选处理，可静默或提示用户）。

App 端收到 `audio_delta` 后解码为字节数组，依次送入 `AudioTrack` 或 `ExoPlayer` 播放；用户发送新消息或点击停止时，应清空播放队列。

### 9.3 音色

`voice` 字段当前支持：

| voice | 说明 |
|---|---|
| `calm_female` | 平静、清晰的女声（默认） |
| `energetic_male` | 活力、友好的男声 |
| `gentle_female` | 温柔的女声 |

音色映射由后端 `TTSAdapter` 维护，转换为 Qwen3-TTS 的 `instructions` 参数。

## 10. 当前限制

- 当前是 Demo 级数据集，不连接真实库存、真实订单和真实支付。
- API Key 只能通过环境变量注入，不能写进代码或提交到仓库。

## 11. Session 与购物车持久化

默认情况下 session 上下文和购物车保存在内存中，服务重启后丢失。如需磁盘持久化，启动前设置环境变量：

```bash
export SHOPGUIDE_SESSION_DIR="data/sessions"
export SHOPGUIDE_CART_PATH="data/carts.json"
```

- `SHOPGUIDE_SESSION_DIR`：每个 session 会生成一个独立的 JSON 文件，存放在该目录下。
- `SHOPGUIDE_CART_PATH`：所有购物车数据写入单个 JSON 文件。

若不设置这两个变量，后端保持原有的纯内存行为，对现有测试和开发流程无影响。
