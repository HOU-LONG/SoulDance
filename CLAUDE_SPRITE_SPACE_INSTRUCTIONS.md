# Claude Code：精灵空间持续迭代工程说明

## 使用方式

1. 把本文件放在 Android 项目根目录。
2. 确认素材包已解压在项目根目录，且某个目录中存在：
   - `manifest/asset_manifest.json`
   - `manifest/avatar_state_matrix.json`
   - `layout/sprite_space_layout.json`
   - `reference/design_reference.png`
   - `AGENTS.md`
3. 启动 Claude Code 后，先让 Claude 阅读本文件和素材包文档。
4. 然后粘贴 `CLAUDE_GOAL_COMMAND.txt` 中的完整 `/goal` 命令。

---

# 1. 产品最终目标

实现一个“换装养成游戏式体验、智能导购 Agent 式功能”的 Android 精灵空间。

用户进入页面时看到一个温暖、游戏化、可培养的女性导购精灵。用户不是面对传统电商商品瀑布流，而是在精灵空间中：

- 和精灵对话；
- 让精灵帮自己寻找商品；
- 看到精灵进入倾听、思考、翻包搜索、展示商品、庆祝和错误反馈等状态；
- 查看从购物包方向出现的商品卡片；
- 对比商品、查看详情、加入购物车；
- 通过互动、任务和购物行为提高亲密度；
- 切换导购服装和职业形象。

第一阶段使用 2D 状态图，但工程边界必须允许以后只替换 `AvatarStage` 为 3D，而不重写页面和业务逻辑。

# 2. 开发优先级

按以下顺序持续推进，前一阶段未稳定前不要堆叠后续复杂功能：

## P0：工程审计和可编译基线

- 确认模块、包名、导航、Compose 版本、状态管理方式。
- 搜索并理解现有 RealtimeEvent、商品模型、网络层和购物车。
- 执行一次基线构建并记录结果。
- 读取素材 manifest、状态矩阵、布局文件和缺失素材说明。

## P1：真实素材静态页面

- 将 Android-ready 资源合并到正确 res 目录。
- 建立统一素材 resolver。
- 实现背景、顶部、舞台、资料区、底部入口和任务栏。
- 接假 UI state。
- 完成 360、393、412dp Preview。
- 编译通过。

## P2：人物状态机和动画

- 接入全部现有人物状态。
- 统一人物锚点。
- 实现待机、搜索、展示、庆祝和错误反馈动画。
- 检查光效位置、zIndex 和性能。

## P3：现有后端事件接入

- 复用真实输入和实时事件流。
- 建立 RealtimeEvent → AvatarVisualState/UI state 映射。
- 流式更新气泡和商品数据。
- 不在 Composable 中调用网络。

## P4：商品与购物车闭环

- 显示动态商品卡片。
- 实现详情、对比和加购。
- 复用已有购物车。
- 成功/失败状态完整。

## P5：换装和亲密度

- 至少支持 DEFAULT、HOME_ADVISOR。
- 保存选择。
- 状态缺失时正确 fallback。
- 亲密度进度动态显示和动画更新。

## P6：视觉和质量迭代

- 真机/模拟器截图对比。
- 修正大小、留白、锚点、层级、触控区域。
- 补测试、testTag、无障碍和性能问题。
- 清理重复代码与 Magic Number。

# 3. 每轮自主迭代循环

每轮都按下面闭环执行：

1. **检查**：读取当前代码、构建状态和上一轮差距。
2. **选择**：只选择最高优先级、能减少验收差距的一组问题。
3. **实现**：小步修改，优先复用现有架构。
4. **验证**：运行格式检查、编译、相关测试；能启动设备时截屏。
5. **比较**：对照设计参考、manifest、layout 和验收标准。
6. **修复**：若验证失败，继续定位，不把失败留给下一轮。
7. **记录**：在回复中给出可被 `/goal` evaluator 看到的明确证据。
8. **继续**：只要还有验收项不满足，就进入下一轮，不等待用户继续提示。

---

# 4. 证据要求

Claude Code 的 goal evaluator 不能自行读取文件或执行命令，因此每轮必须在回复中主动写明：

```text
本轮构建：
- 命令：
- 退出码：
- 关键结果：

本轮测试：
- 命令：
- 通过/失败数量：

本轮视觉验证：
- IDLE 截图/Preview：
- SEARCHING 截图/Preview：
- PRESENTING 截图/Preview：
- 发现并修复的问题：

验收进度：
- [x] 已满足
- [ ] 未满足

下一轮：
- 最高优先级问题
```

不能只写“已经测试”或“应该可以”。

# 5. 视觉判定原则

目标不是复制每一个像素，而是达到以下体验：

- 页面第一眼像完整的精灵养成空间，而不是普通聊天页。
- 房间背景、人物、左右道具、气泡、资料和底部入口形成稳定层级。
- 人物始终是视觉中心，全身完整，脚底稳定。
- SEARCHING 明确表现“人物正在包中找商品”。
- PRESENTING 明确表现“商品被精灵找出来并展示”。
- UI 文本和数据清晰，不与美术素材冲突。
- 页面在不同手机宽度上保持相同视觉重心。
- 任何精灵状态都不能出现白底、灰底或棋盘格方框。

---

# 6. 不要做的事情

- 不要把参考设计图直接放进 `Image(fillMaxSize())`。
- 不要为了像参考图而把动态文字做成图片。
- 不要重新实现已有购物车和网络客户端。
- 不要把后端请求写进 Composable。
- 不要同时大改架构、网络和 UI，导致无法定位问题。
- 不要在没有构建证据时声称完成。
- 不要因为某一个素材状态缺失而停止全部工作，应按 fallback 实现并报告。
- 不要在素材已经存在时重新生成相似图。
- 不要为了“先跑起来”长期保留不可维护的绝对坐标堆叠。

---

# 7. 最终完成报告

只有所有目标满足后，输出：

1. 最终页面入口路径；
2. 新增和修改文件清单；
3. 素材状态映射；
4. 后端事件状态映射；
5. 导购到商品到购物车的完整流程；
6. 换装和亲密度实现；
7. 构建、测试和设备验证证据；
8. 三张关键截图/Preview；
9. 已知非阻塞限制；
10. 后续将 2D 替换为 3D 的接口位置。

不要把未来计划混同于已经完成的功能。
